Overview

This section presents a walkthrough of what must be done by a third party app developer to access the Audible APIs. Some API endpoints are open and can be accessed without authentication. For example, the following command will retrieve the list of audiobook categories available in the Audible catalog: 

$ curl https://api.audible.com/1.0/catalog/categories

Other API endpoints are secured because they depend on user data so callers must authenticate and prove that they have the user's permission. This is done using Login With Amazon (LWA).

What is LWA?

LWA is Amazon's implementation of OAuth2. With LWA, a third party app requests a user's permission to access that user's account by having the user log in to Amazon (to authenticate the user) and accept the permissions the app is requesting (to authorize the app). Once the user grants permission, LWA issues an access token to the app. The app can use the access token in API calls to prove that it was granted access to the endpoint and identify which user's account is being accessed.

How to Use LWA

Complete LWA documentation can be found at the LWA Homepage.

In order to use LWA, you must:

  1. Register as an Amazon Developer from the Amazon Developer Homepage.

  2. Create a security profile from the Amazon Developer Console.

  3. Enable LWA for your security profile.

Follow the LWA documentation to get the access token needed to make calls to secure Audible API endpoints. Here are a few things you will need:

  • Get your Client ID and Client Secret from your security profile.

  • One or more scope values from the table in the next section.

Scopes for Audible API

Audible API resources are limited by OAuth scopes. When you request an access token from LWA, you must indicate which scopes you anticipate using. The following table lists the Audible API scopes and the resources they guard.

Scope Resource Description
audible:annotations.read GET /1.0/annotations Permission to view your Audible audiobook bookmarks and current position information.
audible:catalog.read GET /1.0/catalog Permission to browse Audible's catalog.
audible:content.read GET,POST /1.0/content Permission to play your Audible audiobooks.
audible:browse.read GET /1.0/recommendations Permission to access personalized browsing of the Audible catalog.

Making Calls to Audible API

Once you have an access token, you can make calls to secure Audible API endpoints.

To make secure API calls, just include an HTTP header containing: 

Authorization: bearer ACCESS_TOKEN

where you replace ACCESS_TOKEN with the content of the access token.

LWA Example

Listing product recommendations requires authentication on behalf of an Audible customer. This resource requires the audible:browse.read OAuth scope. To access recommendations, one must request an access token from LWA with that scope and the application's Client ID and Client Secret. The token is placed into the Authorization header and the recommendations resource is requested. This command (when used with a valid access token) will retrieve the first five recommendations for the customer account associated with the access token: 

$ curl --header "Authorization: bearer Atza|IQEBLzAtAhUAkl...iRVFAr4pqVHc" \
"https://api.audible.com/1.0/recommendations?\
response_groups=product_desc&\
num_results=5"

Troubleshooting

If you get an HTTP 403 response the problem could be:

  • An expired access token.

  • A missing or badly formatted Authorization header.

  • The access token's scope does not have permission to access the endpoint.
